# 私募跟车机器人

> 目标：把“根据基金持仓、周报和市场状态判断当前是否适合买/卖”的过程，做成一个可解释、可复盘、半自动运行的研究机器人。

---

## 1. 先定义边界

这个系统不是自动交易系统，至少在第一阶段不是。

它更准确的定位是：

- 一个`基金跟踪与决策辅助系统`
- 一个`周度运行、月度给出赎回建议`的机器人
- 一个`先规则判断，再由 LLM 解释和补充`的系统

对于“每周可申购、每月可赎回一次”的私募证券基金，真正有价值的不是实时盯盘，而是：

- 识别基金当前风格和仓位
- 判断市场是否在给这类风格顺风
- 判断基金是否已经过热、失真或风格漂移
- 在月度赎回窗口前，提前给出`继续拿 / 准备赎回 / 已该赎回`

---

## 2. 这个机器人要回答的核心问题

系统每次运行，最终只回答四个问题：

1. `现在能不能买`
2. `现在适不适合继续拿`
3. `下一个月度赎回窗口前，要不要准备赎回`
4. `为什么`

输出必须是结构化结论，而不是纯聊天：

- `BUY`
- `SMALL_BUY`
- `HOLD`
- `WATCH_REDEEM`
- `REDEEM`

每个结论都要带：

- 置信度
- 触发依据
- 反证条件
- 下次复查时间

---

## 3. 最小可用输入

你前面提到的三类资料，已经足够支撑 MVP：

### 3.1 基金历史净值

用途：

- 计算历史收益、最大回撤、回撤修复速度
- 判断这只基金是趋势型、反转型还是高波动型
- 识别它在不同市场环境里的表现弹性

最低要求：

- 日期
- 单位净值或累计净值

### 3.2 最近几个月的大致持仓

用途：

- 识别基金当前风格暴露
- 判断是否集中在 AI、半导体、医药、消费、红利、周期等方向
- 判断持仓集中度和风格漂移

最低要求：

- 前十大持仓，或行业占比
- 仓位高低
- 数据对应月份

### 3.3 基金运作周报

用途：

- 提取基金经理近期观点
- 判断是否在加仓、减仓、切风格
- 捕捉“市场没变，但经理判断变了”的信号

最低要求：

- 日期
- 本周主要操作
- 市场判断
- 风险提示

## 4. 系统架构

建议按五层做，不要一开始把 LLM 放在核心决策位。

### 4.1 数据输入层

负责接收和整理原始资料。

输入来源：

- 手工上传净值表
- 手工上传周报 PDF / 图片 / Markdown
- 手工录入前十大持仓或行业权重
- 自动抓取市场数据

这一层先允许人工录入，因为私募数据天然不标准，早期最贵的是“拿到对的数据”，不是“全自动抓”。

### 4.2 标准化层

把非结构化资料变成统一字段。

需要产出的对象：

- `FundSnapshot`
- `HoldingSnapshot`
- `WeeklyReportSnapshot`
- `MarketRegimeSnapshot`
- `DecisionSnapshot`

核心标准化动作：

- 把持仓股票映射到行业、主题、风格因子
- 把周报里的自然语言映射成标签
- 把净值序列转成回撤、斜率、波动、胜率等特征

### 4.3 信号层

这里是系统核心，必须是规则引擎，不要直接让大模型“猜”。

至少要拆成五组信号：

1. `基金质量信号`
   - 中长期收益
   - 最大回撤
   - 回撤修复能力
   - 风格稳定性

2. `当前风格顺风信号`
   - 基金重仓方向近期是否强势
   - 宏观状态是否支持该方向
   - 风格是否由政策或流动性驱动

3. `持仓结构信号`
   - 集中度是否过高
   - 是否切换到陌生风格
   - 是否出现“涨出来的好看净值”

4. `位置与节奏信号`
   - 净值是否接近阶段高点
   - 近 4 周是否涨幅过快
   - 是否已经进入不适合追涨区间

5. `流动性与卖点信号`
   - 距下一个月度赎回窗口还有几天
   - 是否需要提前申报
   - 当前风险是否值得等到下个窗口

### 4.4 决策层

这一层把多个信号压成行动建议。

建议第一版采用`评分 + 风险闸门`：

```text
总分 = 0.30 * 风格顺风
     + 0.25 * 基金质量
     + 0.20 * 持仓结构健康度
     + 0.15 * 净值位置
     + 0.10 * 流动性时点
```

再叠加硬性闸门：

- 若出现严重风格漂移，禁止新买
- 若出现高位加仓且市场开始转弱，进入 `WATCH_REDEEM`
- 若月度赎回窗口临近且得分持续低于阈值，升级为 `REDEEM`
- 若净值从阶段高点回撤超过设定阈值，直接触发减分或赎回建议

### 4.5 解释与报告层

LLM 在这里发挥作用，但它不负责最终拍板。

它负责：

- 解释为什么是 `BUY/HOLD/REDEEM`
- 总结周报变化
- 指出和上周相比，什么证据变了
- 生成人能快速阅读的 Markdown 报告

---

## 5. 决策逻辑怎么设计

这个系统最关键的不是“预测涨跌”，而是把你真正关心的决策拆开。

### 5.1 买入逻辑

`BUY` 需要同时满足：

- 基金风格与当前市场顺风
- 持仓没有明显过热
- 经理观点和仓位没有失真
- 当前距离最近高点不太远但也不是追涨末端
- 没有触发风险闸门

`SMALL_BUY` 适用于：

- 大方向顺风
- 但位置一般，或市场波动偏大
- 只允许小仓试探

### 5.2 持有逻辑

`HOLD` 适用于：

- 基金风格仍然有效
- 周报没有恶化
- 回撤仍在正常范围
- 下一个赎回窗口不需要提前动作

### 5.3 卖出逻辑

你的场景里，卖出比买入更重要，因为赎回是月频。

`WATCH_REDEEM` 适用于：

- 风格开始逆风
- 净值位置偏高
- 市场波动上升
- 但还没有到必须立刻赎回的程度

`REDEEM` 适用于：

- 风格明显逆风且经理未调整
- 连续数周周报恶化
- 净值从阶段高点明显回撤
- 距月度赎回申报截止日很近

这里要特别做一个`赎回倒计时模块`：

- `T-7`：开始提示下一个赎回窗口
- `T-5`：锁定本月是否考虑赎回
- `T-3`：若信号恶化，强提示人工确认

## 6. 为什么不能让 LLM 直接做判断

因为这个任务里，最容易出错的是：

- 混淆基金风格
- 过度相信周报措辞
- 忽略月度赎回节奏
- 把“讲得好”当成“该买”

所以职责要拆开：

- `规则引擎`负责打分和闸门
- `LLM`负责解释、摘要、指出变化

这也是你这个仓库现有思路的延续：

> 把叙事写成规则，把规则落实为动作。

## 7. 建议的 MVP 形态

第一版不要做网页，不要做 App，不要接券商接口。

直接做成一个本地运行的研究机器人最合适。

### 7.1 MVP 输入方式

- 把基金净值历史放在 `data/funds/<基金名>/nav.csv`
- 把周报放在 `data/funds/<基金名>/reports/`
- 把持仓快照放在 `data/funds/<基金名>/holdings/`

### 7.2 MVP 输出方式

每次运行输出一份 Markdown：

- 当前结论
- 本周分数变化
- 主要证据
- 风险闸门
- 下个赎回窗口建议

输出示例：

```text
结论：WATCH_REDEEM
置信度：0.74
原因：
1. 基金重仓 AI 硬件，近两周板块强度下降
2. 周报显示仓位仍高，未明显减仓
3. 净值接近阶段高点后回撤扩大
4. 距下个月赎回申报截止仅 4 天
```

### 7.3 MVP 技术选型

结合当前仓库已有 Python 工具，建议第一版统一用 Python：

- `Python`：数据清洗、规则引擎、报告生成
- `Markdown`：输出研究报告
- `unittest`：做最小单元测试
- 后续如要页面，再加轻量 Web 层

## 9. 实施顺序

建议按四步推进。

### 第一步：先把数据喂进去

目标：

- 一只基金能完整录入
- 能读净值、持仓、周报
- 能生成标准化快照

如果这一步做不好，后面所有判断都会失真。

### 第二步：先做规则，不接 LLM

目标：

- 先跑出稳定分数
- 明确哪些信号有效，哪些只是噪音
- 建立买卖建议阈值

这一步必须可回测、可解释。

### 第三步：再接 LLM 做摘要和解释

目标：

- 自动读取周报
- 归纳基金经理观点变化
- 自动生成中文报告

但最终建议仍由规则引擎给出。

### 第四步：做回测和周度运行

目标：

- 回看过去 1 到 3 年
- 评估这个机器人在不同市场环境下的表现
- 看它是否真的提高了“月度申赎决策”的胜率

---

## 10. 回测时必须验证什么

这个系统不是看“收益率最高”，而是看它是否真的改善了决策质量。

至少验证：

- 机器人建议买入后，未来 1 个月胜率
- 机器人建议赎回后，是否有效避开回撤
- 高波动市场里，是否减少了追涨买入
- 基金风格漂移时，是否能提早识别

更重要的是看：

- `胜率`
- `盈亏比`
- `最大回撤改善`
- `无效交易减少了多少`

---

## 11. 风险与现实约束

要提前接受三件事：

### 11.1 私募数据不标准

周报、持仓、净值披露口径可能不统一。

所以系统必须支持：

- 手工修正
- 覆盖字段
- 保留原始文档和解析结果

### 11.2 月度赎回不是实时交易

所以系统核心不是“抓最高点”，而是：

- 提前识别是否需要进入赎回准备状态
- 避免在明知转弱时错过窗口

### 11.3 这不是预测系统，是决策支持系统

系统的价值不在于每次都猜对，而在于：

- 少犯明显错误
- 少追高
- 风险变坏时早点提醒
- 所有动作都能复盘

## 12. 我建议的第一版目标

如果只做一个最小版本，我建议目标定成：

> 输入一只基金近 2 年净值、近 3 个月持仓、近 8 期周报，输出一份“本周是否适合买/继续拿/准备赎回”的 Markdown 报告。

只要第一版能稳定完成这件事，这个系统就已经有用了。
